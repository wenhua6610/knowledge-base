## 调 PE 的六步走

这个流程是老师说的"面试标准答案"，逻辑挺清晰的：

1. **明确理想态目标** — 先定好最终结果长什么样，不要边做边想
2. **测试模型能力边界** — 看现有模型能不能达到要求，探明上限
3. **写初版 Prompt（V1）** — 用六要素框架：角色 / 背景 / 目标 / 要求 / 上下文 / 示例
4. **小批量测试** — 10\~20 条，要有多样性，不能全是相似的输入
5. **定向调整** — **每次只改一个点**，改完跑 10 条验证，验证完再跑 5 条
6. **批量验收上线** — 跑 100 条看稳定性，达标后小范围上线 → 再全量

**关键点：渐进式修改，每次只调一处。** 这个原则在其他地方也反复出现，是调 PE 最容易犯的错——一次改三个地方，根本不知道效果来自哪里。

## SP 和 PE 的区别

之前这两个概念有点混。课里讲得比较清楚：

- **SP（System Prompt）**：内嵌在产品/模型里，每次都先跑一遍，权限更大，需要通用性
- **PE（User Prompt）**：用户输入的那个，针对具体任务

比如 Cursor 会自动做 todo list 再分步执行，是因为它的 SP 里写了「代码任务执行规则」。角色扮演产品说话有固定风格，是因为 SP 里写了人设。

**写 SP 要注意**：权限大，但不能太局限，要能适配所有任务场景。

## Workflow：节点 + 接力棒思维

工作流的本质是把重复的复杂任务放到一个可反复使用的流程里。

关键概念：**每个节点都有 Input 和 Output，后面节点的 Input 可以引用前面任意节点的 Output，不限于上一个。**

老师把这个叫"接力棒思维"——传什么，接什么，传给谁，想清楚再搭。

**三个工作流工具对比：**

| 工具 | 难度 | 适用场景 |
|------|------|---------|
| Coze（扣子） | 最简单，低代码 | C 端日常使用 |
| Dify | 中等，需要代码能力 | B 端，可本地部署 |
| N8N | 最难，高代码 | B 端专业场景 |

三个工具原理相通，都是节点思维，学会 Coze，其他的逻辑也能理解。

## 项目案例：文生图 Prompt 扩写工作流

这个案例记得比较完整，演进路线值得参考。

**问题背景**：文生图产品需要训练 Caption 模型，数据格式是「用户原始 prompt + 改写后的 prompt」。非专业人员词汇量不够，写出来的改写 prompt 太差。

**方案一：通用 PE**
→ 写一个通用 PE 对用户 prompt 进行扩写
→ 问题：结果呆板、固化、缺乏多样性，不同场景（人像/动物/风景/设计）全部套同一套词

**方案二：分场景意图分类**
→ 打捞 1000 条线上数据分析，发现绘画/摄影/设计三类占 90%
→ 工作流加一个**意图识别节点**，根据 prompt 自动分到对应场景
→ 每个场景的 PE 各有侧重：绘画类看风格/笔触，摄影类看构图/光影/4K，设计类看排版/字体
→ 问题：非专业人员拿到改写 prompt 还是不知道好不好

**方案三：加前后对比图**
→ 用原始 prompt 生一张图（before），用改写后的 prompt 再生一张（after）
→ 非专业人员眼睛看，有增益就说明改写质量好
→ 落地成果：标注准确率 80% → 96%+，单人产能 80 条/天 → 150 条/天

**这个演进思路本身就是一种答题模板**：方案一→遇到问题→方案二→遇到问题→方案三，每次迭代都有具体的问题和数据。
