上了一门讲智能客服数据构建的课，4 小时 8 分钟，核心问题是：没有原始数据，怎么从 0 到 1 构建 10 万条高质量问答对。

---

## 自动化实施五步法

这个框架适用于所有自动化项目，老师说是"所有自动化项目的脚手架"：

1. **明确目标** — 背景、目标、数据是什么
2. **识别瓶颈** — 哪里效率低，哪里准确率不够
3. **设计方案** — PE / Workflow / 脚本，选合适的
4. **落地验证** — 数据说话，不要全是 5 的倍数
5. **项目复盘** — 沉淀 badcase，优化流程

**关于数据真实性**：简历里不要写 80%、85%、90% 这种整数。真实工作中准确率都带小数点。写 99.67%、76.6%、35% 才显得真实。

---

## PE / Workflow / 脚本，怎么选？

| 工具 | 适合场景 | 最大限制 |
|------|---------|---------|
| PE | 简单轻量任务 | 效果天花板 = 模型上限 |
| Workflow | 中等复杂度、有流程的任务 | 只能用节点里有的功能 |
| 脚本 | 大批量、复杂任务 | 需要会写（或让 AI 写）代码 |

脚本最灵活，写一次可以无限复用，处理几十万条数据没问题。Workflow 最大的局限是节点功能有限，脚本可以做任何工作流做不到的事。

**AI IDE 工具**：Trae、Cursor、Claude Code，现在直接告诉它"帮我写一个脚本，调用 API，批量生成 Excel 数据"，它就能跑出来。

## 六个项目的演进逻辑

从 0 构建 10 万条女装智能客服问答数据，老师设计了六个递进项目：

**项目 1：通用 PE + 脚本批量生成**
规则文档 → 脚本 → 批量输出 Excel

关键点：批量生成时**不要给 Few-shot**。你给的样例会成为所有数据的参考，特别容易出现雷同。

**项目 2：子场景细分**
问题：粗分 4 个类别（售前/售中/售后/情绪）直接生成 10 万条 → 大量重复
解决：让 AI 细分 100 个子场景，每个场景生成 N 条 → 多样性大幅提升

**项目 3：用 Skill 改写 Prompt**
问题：数据不够真实，Prompt 太书面化
解决：构建一个"人性化改写 Skill"，按 MBTI/年龄/情绪/修改强度/长度等维度改写

**项目 4：多模型 Response 改写**
同一个 Prompt 喂给 3 个不同人设的模型 → 得到 3 个 Response → 选最优

**项目 5：自动化评测打分**
按 1/2/3 分标准打分，3 分直接保留，2 分优化，1 分改写

**项目 6：RAG 反向构建**
爬真实商品信息（JSON）→ 基于商品信息反推真实用户 Prompt → 多模型生成 Response → 评分选优
结果：一条数据 = 真实商品信息 + 真实用户 Prompt + 优质 Response

## Skill 是什么，怎么用

老师的类比挺好记的：**大模型是手机，Skill 是手机上的 App。**

Skill 的渐进式调用机制：
- 主 SKILL.md 描述"什么时候触发"
- reference/ 文件夹放具体技能模块
- 用户说"做一个 UI 网页" → 模型只调取 UI 和 Icon 模块，其他不打开

好处是省 Token，而且精准。批量生成数据时，让 Skill 负责改写 Prompt 的部分，效果比直接用 PE 好很多。

## 自动化评测的四个坑

这部分信息量很高，之前没有系统想过：

**坑 1：要先讲原因，再打分**
模型的运算逻辑是向后生成的。先打分 → 原因是为了配合分数编出来的；先讲原因 → 分数才是真正分析后的结论。

**坑 2：位置偏见**
模型会自动给靠前的答案打高分（注意力机制偏向前面）。解决：换不同顺序测试，看分数是否稳定。

**坑 3：冗长偏见**
模型会觉得字数多的答案分数高。解决：控制候选答案的字数差异。

**坑 4：自我偏好**
模型偏好同家族模型生成的内容。用 Claude 评测，它会觉得 Claude 生成的更好。解决：用 3\~4 个不同家的模型同时打分，看一致性。

## 人机分工的原则

> 机器干机器擅长的事，人干人特有的事。

- **机器**：重复性、规律性、格式判定、批量处理
- **人**：情感、网感、美感、文化敏感性、安全终审

自动化评测能做到 80\~90% 一致性之后，剩下的 10\~20% 才需要人工介入。这个比例是判断自动化值不值得做的关键。
